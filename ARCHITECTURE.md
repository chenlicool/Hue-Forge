# {{PROJECT_NAME}} 技术架构与实现文档

## 1. 项目概述
本项目当前定位为颜色系统生成工具，运行形态为单页前端应用。

## 2. 目录结构
- 根目录：迁移后的 Vite 工程入口与项目文档。
- `archive/`：历史备份与遗留配置归档目录；当前存放迁移来源压缩包与旧元数据文件。
- `scripts/`：项目辅助脚本目录；当前存放本地颜色对比度计算脚本。
- `src/`：界面组件、状态 Hook、颜色算法工具。
- `src/components/ExportPanel.tsx`：代码导出面板，负责多格式预览与复制。
- `src/utils/exportFormats.ts`：导出文本生成器，负责把当前 palette 转成 `CSS / Tailwind / Tailwind 4 / Tokens`，并按需包装为 AI 友好文本。
- `archive/color-system-v2.0.zip`：迁移来源压缩备份，是否长期保留待确认/未知。

## 4. 核心流程
1. `useColorSystem` 维护色相配置、亮度基线与色阶曲线。
2. `MatrixBoard` 展示基础亮度列与各色相生成结果。
3. `InsightPanel` 基于对比度计算提供曲线调节与可访问性反馈。
4. 工具栏负责预设色相生成与代码导出入口。
5. 颜色生成采用“色相 + 饱和度曲线 + 亮度曲线”的三段式模型：
   `hue` 决定色相，`saturationCurve` 与 `brightnessCurve` 按 step 一一对应，最终逐 step 转为 HSB 颜色。
6. 色板生成时按当前 `baseScale` 长度遍历每个 step，使用 `hsbToHex()` 将 HSB 转为 Hex，并同步计算：
   黑字对比度、白字对比度、对应 WCAG 等级。
7. 预设色相不是统一公式推导，而是按 hue 区间映射到一组人工定义的饱和度/亮度曲线：
   例如 Red、Orange、Yellow、Blue、Indigo 等各自维护独立曲线，再按当前 step 数量做插值缩放。
8. `Gray` 与 `Accent Gray` 属于特殊中性色：
   `Gray` 饱和度固定为 0；
   `Accent Gray` 保留低饱和冷灰倾向，当前默认 hue 为 210，饱和度曲线已压低为弱色相版本。
9. 当用户在 `Base Scale` 轨道点击新增 step 时，系统会按亮度位置插入一个新基线值，并同步扩展所有 hue 的饱和度/亮度曲线，保持数组长度一致。
   当前扩展策略分两类：
   头部插入不再直接做平均，而是对前端一小段曲线做“边缘重分布”，把最浅的 2-5 个点重新拉开，避免新增后 `10` 与 `20` 基本同色；
   同时最浅一阶的 `saturation` 会保留至少 `3%`，避免持续扩阶后 `10` 被完全去色成 `0`；
   其中 `saturation` 在高亮区会额外执行一次更积极的头部去色，避免 24 阶及以上时 `Amber → Blue` 一段的最浅阶显得比其他色条更重；
   中段/尾部插入仍采用局部插值，维持原有中段节奏不被整体改写。
10. 当用户修改单个 step 的亮度基线时，会把亮度差值同步施加到所有 hue 的同索引亮度点，保持整体阶梯结构对齐。
11. 导出功能直接复用当前 palette 计算结果，不重复推导颜色：
   `ExportPanel` 读取当前 `palettes + baseScale`，由 `exportFormats.ts` 一次性生成 `CSS / Tailwind / Tailwind 4 / Tokens` 四类文本。
12. `Tokens` 导出除 Hex 颜色外，还附带 HSB、黑白对比度和 WCAG 结果，便于 AI 或下游工具做二次消费。
13. 导出面板支持 `AI Export` 开关：
    开启后不改变原始数据结构，只在复制内容外层追加格式说明、palette 摘要与使用约束，便于直接粘贴到 AI。
14. 颜色角度映射规则如下，新增 hue 与预设 hue 共享同一套区间判断：
    `345°-15°`：Red 曲线。低亮部保持低到中等饱和度，深色区维持暖红识别，避免过早脏暗。
    `15°-38°`：Orange 曲线。中高亮区域允许快速升饱和，亮部亮度接近 100，保证橙色在浅阶依然鲜明。
    `38°-60°`：Yellow 曲线。黄色天然高亮，算法通过高亮度配合中高饱和，避免浅阶发灰，同时在深阶压低亮度保留层次。
    `60°-115°`：Lime 曲线。亮部维持很高亮度，中段开始快速提饱和，用来保留荧光感与黄绿辨识度。
    `115°-155°`：Green 曲线。饱和度和亮度增长较均衡，目标是获得稳定、通用的绿色层级，而不是极端鲜艳。
    `155°-180°`：Teal 曲线。与 Green 接近，但在中后段保留更高饱和，强化青绿色科技感。
    `180°-195°（含 195°）`：Cyan 曲线。亮部亮度很高，深色段仍保留一定亮度，避免青色在暗部过早失去清透感；195° 归入该段，用于改善 24 Hues 下 Blue 中段可读性。
    `196°-224°`：Blue 曲线。浅阶维持接近白蓝，中段后显著增加饱和与对比，形成典型品牌蓝梯度。
    `225°-250°`：Indigo 曲线。亮部非常亮，中后段迅速加深并提高饱和，用于构建偏紫蓝的强层级色。
    `250°-285°`：Violet 曲线。整体饱和度起点较低，随 step 增加平滑提升，避免紫色在浅阶过脏。
    `285°-315°`：Pink 曲线。亮部接近中性粉，中段进入明显色相区，深段仍保留足够颜色感但避免过荧。
    `315°-345°`：Crimson 曲线。作为洋红与红之间的过渡色，采用比 Red 更偏玫红的深色表现，保持冷暖之间的张力。
15. 当前默认预置 hue 采用以下固定角度：
    `Gray = 0°`、`Accent Gray = 210°`、`Red = 0°`、`Orange = 30°`、`Yellow = 45°`、`Lime = 90°`、
    `Green = 140°`、`Teal = 170°`、`Cyan = 190°`、`Blue = 210°`、`Indigo = 240°`、
    `Violet = 260°`、`Pink = 300°`、`Crimson = 330°`。
16. 各颜色模板的核心数组特征如下：
    Red 当前改为更明确的“低起点 -> 中段持续抬升 -> 尾段回落”饱和度曲线，用于贴近用户指定的品牌红走势；
    Crimson / Pink / Violet 仍保持偏低起点、后段增强，用于避免浅色脏感。
    Orange / Yellow / Lime 一组亮部极亮、局部高饱和，用于保持暖色与荧光色的明快感；本轮已轻微压低 Lime 中段亮度以稳定 Yellow 60 的白字可读性。
    Green / Teal / Cyan 一组亮度下降更平滑，强调中段到暗段的稳定识别；本轮已压低 Cyan 中段亮度以覆盖 Light Blue / Blue 的 60 阈值。
    Blue / Indigo 一组在中后段快速拉开亮度与饱和差，形成更强品牌色阶层次；本轮已压低 Blue 与 Indigo 的 60/70 中段亮度以满足白字阈值。
    Gray 通过零饱和度生成纯中性灰；Accent Gray 通过低饱和冷灰曲线提供带轻微冷调的中性色替代。

## 5. 关键设计决策
- 当前采用纯前端本地计算，不依赖后端接口。
- 迁移策略已落地为“提升根目录运行入口并验证通过后，清理历史源码目录”。
- 导出能力采用“当前视图即当前导出”原则：
  不额外维护第二套 token 状态，避免 UI 调整结果与导出结果不一致。
- 颜色空间当前采用 HSB 而非 Lab/OKLCH：
  优点是实现简单、交互直接、与 hue 手柄模型一致；
  代价是低饱和中性色区域的感知均匀性有限，`Accent Gray` 需要单独曲线修正。
- 色板算法以“人工曲线模板 + 插值缩放”为主，而不是纯数学自动生成：
  这样可以分别控制不同色相在亮部、中段、暗部的饱和度与亮度表现。
- 对比度评估对每个 step 同时计算黑底/白底可读性，并输出 `AAA` / `AA` / `Fail`：
  右侧面板的安全区与明度调节依赖同一套对比度计算结果。
- hue 改名规则与颜色生成解耦：
  普通色相拖动 hue 时会根据角度区间自动命名，`Accent Gray` 通过 `customName` 固定名称，不随拖动自动改名。

## 6. 风险与约束
- 现有界面文案国际化与设计 Token 规范存在待确认/未知项，本次以迁移可运行性优先。
- 原始子项目中的 AI Studio 配置与当前本地运行关系待确认/未知。
- `archive/metadata.json` 为历史遗留元数据文件，当前运行链路未引用；是否可删除待确认/未知。
- 当前 step 编号仍按 `10, 20, 30...` 线性展示；当 step 数量不等于 10 时，显示编号与实际插值密度并非严格语义等价，这是现状约束。
- Tailwind 导出的 key 采用当前 palette 名称的 kebab-case 结果；如果后续允许更自由命名，需评估命名合法性与冲突处理。
